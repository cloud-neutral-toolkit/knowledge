# Ansible

Ansible 是 Cloud-Neutral Tool Map 中交付与变更控制（IaC & DevOps）这一层里，最“老派”、却依然最具落地能力的自动化工具之一。相比 Puppet / SaltStack 更偏系统工程的设计，Ansible 可以无压力地从管理1台机器起步，逐步扩展到成百上千台。在 Kubernetes、GitOps、IaC 成为主流之前，它就已经在解决一个非常现实的问题：

> **如何把系统状态，可靠地推到“应该长成的样子”。**

---

## 项目主要特性

- 声明式 + 任务驱动 自动化模型  
- 基于 **SSH / WinRM，无需 Agent  
- 覆盖 OS / 中间件 / 网络 / 云资源
- 大量成熟模块，适配传统与云环境  
- 与 CI/CD、IaC 工具天然互补，而非竞争
- 
Ansible 的定位从来不是“持续对账”， 而是：**把一次变更，稳定地执行完成。**

---

## 优缺点

| 优点 | 局限 |
|---|---|
| 上手成本低，可读性强 | 非持续收敛模型 |
| 覆盖面极广，落地能力强 | 复杂 Playbook 可维护性下降 |
| 对传统环境 / 混合环境极友好 | 大规模并发效率有限 |
| 不绑定云厂商或平台 | 不擅长处理长期状态漂移 |

---

## 适用场景

| 适合 | 不适合 |
|---|---|
| OS 初始化、基线配置 | 强实时、持续对账场景 |
| 中间件 / 网络设备配置 | 期望自动纠偏的系统 |
| 混合云 / 传统 IDC | 纯声明式 GitOps 模型 
| 与 Terraform / Argo CD 配合 | |
---

## 工程判断

Ansible 并不是 GitOps 的替代品，也不是 IaC 的竞争者。它清晰地站在 “一次性交付与变更执行” 这一工程责任边界上：**当你需要“把事干成”，而不是“持续盯着状态”，Ansible 依然是最可靠的选择之一。**

在现代体系中，它更适合的位置是：

> Terraform 定义资源， **Ansible 完成系统与软件层的落地**， Argo CD 负责长期状态收敛。
